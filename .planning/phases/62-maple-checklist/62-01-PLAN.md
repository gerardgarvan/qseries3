**Fix 1: theta2/3/4 must honor their q argument.**

Currently all three functions have `const Series& /*q*/` — the q param is commented out and ignored. Fix each to detect the q-power from the first argument and scale exponents accordingly.

For `theta3(const Series& q_arg, int T)`:
- Detect k: if q_arg has exactly one term at exponent k with coefficient 1 (i.e., q_arg represents q^k), use k as the scaling factor. Otherwise k=1 (standard q).
- Loop condition: `n * n * k < T` (was `n * n < T`).
- Set exponent: `exp = n * n * k` (was `exp = n * n`).

For `theta2(const Series& q_arg, int T)`:
- Same k detection.
- Scale q_shift: `result.q_shift = Frac(k, 4)` (was `Frac(1, 4)`).
- Loop condition: `n * (n + 1) * k < T` (was `n * (n + 1) < T`).
- Set exponent: `exp = n * (n + 1) * k` (was `exp = n * (n + 1)`).

For `theta4(const Series& q_arg, int T)`:
- Same k detection.
- Loop condition: `n * n * k < T`.
- Set exponent: `exp = n * n * k`.

Helper to extract k from q_arg:
```cpp
int k = 1;
if (q_arg.c.size() == 1) {
    auto it = q_arg.c.begin();
    if (it->second == Frac(1) && it->first > 0)
        k = it->first;
}
```

**Fix 2: tripleprod and quinprod T parameter semantics.**

Currently both use T as both the loop iteration bound (`for (int n = 1; n < T; ++n)`) AND the truncation (`Series::one(T)`, `truncTo(T)`). This means `tripleprod(q, q^3, 10)` produces output truncated at O(q^10) instead of computing enough factors for terms up to O(q^{3*T^2}).

Fix approach: T remains the truncation order of the output (consistent with all other qseries3 functions). But the loop must compute enough factors. Determine max iterations by checking when factor exponents exceed T.

For `tripleprod(const Series& z, const Series& q, int T)`:
- Keep `result = Series::one(T)`, `truncTo(T)`.
- Change loop: `for (int n = 1; ; ++n)`. The three factors at iteration n involve exponents from z * q^(n-1), z^(-1) * q^n, and q^n. For z = q^a and q = q^b: the minimum new exponent in factors is b*n. Break when b*n >= T (all new factors are 1 mod O(q^T)).
- To determine the break condition without assuming z and q are pure q-powers: compute the lowest exponent of `qt.pow(n)`. If it's >= T, all subsequent factors are trivially 1 modulo O(q^T), so break. A safe bound: find the minimum non-zero exponent `qmin` in q (typically 1 for q or k for q^k). Break when `qmin * n >= T`.

For `quinprod(const Series& z, const Series& q, int T)`:
- Same approach for the first loop (n loop).
- Second loop (m loop for z^2 factors): `for (int m = 1; ; ++m)`. Break when `qmin * (2*m - 1) >= T`.

**Fix 3: winquist T semantics check.**

The winquist function uses `3 * n * (n + 1) / 2 < T` for the outer loop and `exp < 0 || exp >= T` to skip terms — this is already treating T as truncation order, which is correct. However, the inner loop `for (int m = -T; m <= T; ++m)` may be too aggressive. Verify that the implementation produces correct output for the Maple examples. The inner loop bound should be tightened: `m*(3*m+1)/2 < T` would suffice, or leave as-is since terms with exp >= T are already skipped.

Actually, winquist has a deeper issue: it multiplies a, b, q series with various powers inside the loop, treating them as formal power series. This is the correct approach if a, b, q are concrete series (like q^5, q^3, q^11). The T parameter is already used as truncation. The issue is only that we need large enough T. This is fine — the test script will pass appropriate T values.

After all fixes, rebuild: `g++ -std=c++20 -O2 -static -o qseries src/main.cpp`

Quick smoke test: `echo "theta3(q^2, 100)" | ./qseries` should NOT equal `echo "theta3(q, 100)" | ./qseries`. And `echo "tripleprod(q, q^3, 100)" | ./qseries` should show terms up to q^57 or higher.
